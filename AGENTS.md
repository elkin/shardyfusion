# Repository Guidelines

`shardyfusion` is a sharded snapshot writer/reader for S3-compatible object storage.

Keep this file short and accurate. It is an operating guide for agents, not a
replacement for the MkDocs site. If a code change affects architecture, module
organization, CLI commands, public API, important behavior, or developer
workflow, update this file when needed and record breaking or otherwise
important changes in `CHANGELOG.md`.

## Development Workflow

Use `just` for repo workflows because recipes set the right extras, tox envs,
and environment flags. Run `just --list` for the current recipe list.

Common recipes:

- Bootstrap: `just setup`
- Health check: `just doctor`
- Dependency sync: `just sync`
- Auto-format/lint-fix: `just fix`
- Quality gate: `just quality`
- Unit tests: `just unit`
- Integration tests: `just integration`
- Container E2E tests: `just d-e2e`
- Full local CI + E2E: `just ci d-e2e`
- Docs build: `just docs`
- Docs live server: `just docs-serve`
- Coverage: `just coverage`

Direct `rg`, `python -c`, or a narrow `pytest` invocation is fine for diagnosis
and tight loops. Prefer starting tests through the matching `just` recipe
because it runs the appropriate tox envs with the repo's extras and environment
setup. This matters especially for Ray, Spark, tox matrix, docs, and container
flows.

Package management is via `uv`, normally invoked by `just`. Optional extras are
explicitly named in `pyproject.toml`; avoid adding implicit backend defaults.

## CI And Environment

- CI stages on push/PR: quality, package, unit, integration.
- E2E is local/container only via `just d-e2e`.
- CI matrix is generated by `just ci-matrix`
  (`scripts/generate_ci_matrix.py`).
- Spark jobs require Java 17. CI uses Temurin.
- Spark 3.5 is not compatible with Java 21 in this repo.
- Dask, Ray, and CEL do not run on Python 3.14.
- Ray tests need `RAY_ENABLE_UV_RUN_RUNTIME_ENV=0`; tox/just set this. Direct
  Ray pytest runs can hang or create unexpected worker environments without it.
- Container recipes use an isolated uv project env at
  `/opt/shardyfusion-venv`.
- Garage E2E uses path-style S3 addressing.

## Architecture Rules

The library has independent writer, reader, CLI, vector, and unified
KV+vector paths that share config, routing, manifest models, storage, metrics,
and core writer logic. See `docs/architecture/index.md` before changing
cross-cutting contracts.

High-risk invariants:

- Writer and reader shard routing must stay identical. Hash routing is
  `xxh3_64(canonical_bytes(key), seed=0) % num_dbs`, implemented centrally in
  `routing.py`. Do not reimplement it in framework-specific code.
- `canonical_bytes()` rules are part of the routing contract: int uses signed
  8-byte little-endian, str uses UTF-8, bytes pass through unchanged.
- Spark, Dask, Ray, Python, vector, and reader routing changes need targeted
  routing contract tests.
- CEL sharding must keep writer-side assignment and reader-side routing in
  agreement. Compiled CEL objects are process-local; use cached compilation
  inside worker-side functions rather than sharing compiled instances across
  worker boundaries.
- Manifests are sparse: only non-empty shards are stored. Reader routing pads
  missing shard IDs with null shard metadata.
- `_CURRENT` publish is two-phase: manifest first, pointer second. `rollback`
  is the CLI command that mutates `_CURRENT`; read selection via `--ref`,
  `--offset`, or REPL `use` is session-local.
- Reader refresh/reference-counting protects borrowed shard readers. Preserve
  borrow/close semantics in sync and async readers.
- Package-specific errors inherit from `ShardyfusionError` and expose
  `retryable`.
- Optional heavy dependencies are lazy-imported. Preserve import-time behavior
  for SlateDB, APSW, obstore range-read VFS, LanceDB, sqlite-vec, and unified
  reader exports.
- Runtime S3/object-store I/O uses `obstore` backends. Do not introduce boto3
  or aiobotocore on hot paths without an explicit design decision.

## Testing Guidance

Use focused tests while developing, but start them with `just` when practical.
The `just unit` and `just integration` recipes run the relevant tox envs instead
of only the current interpreter, so they catch dependency and version issues
that direct pytest can miss. Both recipes accept test paths directly after any
recipe options for subset runs. Run a broader `just` recipe before calling work
complete.

Useful targeted commands:

- `just unit tests/unit/writer`
- `just unit tests/unit/cli`
- `just unit tests/unit/vector`
- `just integration tests/integration/read`
- `just unit tests/unit/writer/core/test_routing_contract.py`

Add or update tests according to risk:

- Routing, sharding, or key encoding: property/contract tests plus affected
  framework tests.
- Writer publish, cleanup, retry, or manifests: unit tests plus integration
  where S3 behavior matters.
- Reader refresh/concurrency/async behavior: reader unit tests and targeted
  integration tests when manifests or storage are involved.
- CLI behavior: Click runner tests under `tests/unit/cli` and docs/reference
  updates when commands or flags change.
- Metrics: `tests/unit/metrics/` and the relevant optional-extra tox env.
- Docs examples or use-case behavior: verify the live API/CLI behavior, then
  run `just docs`.

Before requesting review or saying the branch is ready, run `just ci d-e2e`
unless the user asked for a narrower verification or the environment makes the
full run impractical. If it cannot be run, say exactly what was skipped.

## Coding Style

- Python 3.11+ with full type hints. Ruff and pyright are enforced.
- Follow existing module patterns and local helper APIs.
- Keep Spark-specific logic in Spark modules; Dask/Ray/Python paths should not
  inherit Spark requirements.
- Use dataclasses with `slots=True` for performance-sensitive models when
  consistent with nearby code.
- Manifest models use Pydantic with enum names preserved.
- Preserve deferred imports for optional extras.
- Keep public API changes explicit in `shardyfusion/__init__.py`, docs, and
  changelog when applicable.
- Commit subjects should be concise and imperative. Conventional Commit
  prefixes (`fix:`, `feat:`, `test:`, `chore:`) are recommended.

## PR And Review Checklist

- Keep commits scoped by purpose.
- Include tests for functional changes.
- Update docs when behavior, public API, config, or CLI usage changes.
- Update `AGENTS.md` only when agent-facing guidance changes.
- Update `CHANGELOG.md` for breaking or important user-visible changes.
- In PR notes, include what changed, why, impacted area, and commands run.

## Gotchas

- Python writer parallel mode uses multiprocessing `spawn`; worker inputs must
  be picklable.
- Ray writer temporarily changes `DataContext.shuffle_strategy` and restores it
  under a lock.
- `TokenBucket` is not thread-safe; use `ThreadSafeTokenBucket` when sharing a
  limiter across threads.
- `try_acquire()` must remain non-blocking and safe for async code.
- `acquire_async()` uses `asyncio.sleep`; do not call sync `acquire()` from an
  event loop.
- SQLite range-read VFS requires APSW and uses per-instance VFS names.
- SQLite adapter writes build a local DB and upload it on close; the
  shard's `checkpoint_id` is an opaque `uuid4().hex` stamped by the
  writer (see `shardyfusion._checkpoint_id.generate_checkpoint_id`), not
  a SHA-256 of the materialized DB file. The SHA-256 semantic was
  dropped in the slatedb 0.12 upgrade.
- SQLite adapters emit a `shard.btreemeta` sidecar alongside `shard.db` by
  default (opt out via `emit_btree_metadata=False`). Requires APSW + zstandard
  from the `[sqlite-range]` extra; silently skips when either is missing.
  Format spec: `docs/reference/sqlite-btree-sidecar-format.md`.
- slatedb 0.12 only exposes an async API (`slatedb.uniffi.Db` /
  `WriteBatch` / `DbReader`). shardyfusion bridges sync writer/reader
  Protocols to it through a process-global daemon-thread asyncio loop in
  `shardyfusion._slatedb_runtime`. Public sync→async entry points are
  `shardyfusion._async_bridge.call_sync(coro, *, timeout=None)` and
  `gather_sync(coros, *, timeout=None, return_exceptions=False)`; the
  legacy `run_coro` is a back-compat alias. Both raise
  `BridgeTimeoutError` on timeout and refuse re-entry from the bridge
  loop. Per-call bridge cost is ~15–40 µs; sync iterators must batch via
  `iterator_chunk_size` (default 1024 on `SlateDbReaderFactory`) or
  per-row hops dominate scans by ~30×. For N concurrent shard hops
  prefer `gather_sync`/`SlateDbReaderFactory.open_many` to pay the hop
  cost once instead of N times.
- All `slatedb.uniffi` symbol lookups go through
  `shardyfusion._slatedb_symbols`. Tests should monkey-patch
  `sys.modules["slatedb.uniffi"]` (and `sys.modules["slatedb"]`) rather
  than the top-level `slatedb` module alone.
- `DbAdapter` write-side Protocol uses `seal() -> None`, **not**
  `checkpoint() -> str | None`. The writer stamps `checkpoint_id`
  separately. Custom adapters that still implement `checkpoint()` will
  fail Protocol checks. Both `DbAdapter` and `DbAdapterFactory` live in
  `shardyfusion/_adapter.py` (backend-neutral); `slatedb_adapter.py`
  only contains SlateDB-specific code (`SlateDbFactory`,
  `DefaultSlateDbAdapter`). Public re-exports under `shardyfusion.*`
  unchanged.
- `SlateDbReaderFactory` accepts `checkpoint_id` for Protocol symmetry
  but ignores it (slatedb 0.12 has no read-side checkpoint API). Cache
  identity for SlateDB shards is the `db_url` only; SQLite/SQLiteVec/
  LanceDB factories still key on `checkpoint_id`.
- Performance microbenchmarks (`tests/integration/perf/`,
  `@pytest.mark.perf`) are excluded from default `pytest tests` via
  `addopts = "-ra -m 'not perf'"`. Run them via `just perf`. They are
  not part of `just ci`.
- Session-scoped Spark and S3 fixtures are shared across tests for speed.
- Writer scenario imports in `tests/helpers/` are intentionally deferred so
  reader-only collection does not require writer extras.
